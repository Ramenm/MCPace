# MCPace assurance review

Generated by `npm run assurance`. This report answers: what is smart/useful, what is proven, what is still unproven, and how to review the product without fooling yourself.

- Overall: **needs-live-rust-proof**
- Claims: 12 pass, 2 warn, 0 fail

## How to look at the product

- **What should a normal user see?** One readiness verdict, endpoint, server launch command/URL, evidence state, and one safe next action.
- **What should an operator see?** Per-server lane, blockers, safeguards, recommended policy, command runbook, and whether policy drift exists.
- **What should stay hidden by default?** Secret values, raw JSON/logs, manual concurrency controls, and disabled tools represented as usable.
- **What proves the product is smarter?** Backend evidence drives UI decisions; package names alone do not grant trust; runtimeControlPlane ties evidence, risk, isolation, resources, and parallelism into one reviewable model.

## Correct verification flow

1. Static source contract: npm run check:assurance and npm run check.
2. Release contract: npm run build:release-artifacts and inspect zipVerification.status.
3. Platform static contract: npm run check:platform and inspect reports/platform-proof.md.
4. Rust host contract: npm run check:rust and cargo build --release on a machine with Rust 1.95.0.
5. Live user contract: mcpace up, open dashboard, add one harmless server disabled, enable intentionally, run Test once.
6. MCP client contract: point one client at /mcp, run tools/list, then one read-only upstream_call.
7. Security negative contract: bad Origin/Host/content headers/session/request-id tests must fail closed.

## How to identify wrong vs right states

- **Right:** empty install exposes no upstream tools; user adds one server intentionally; dashboard shows command, evidence, and one next action.
- **Right:** ready means live `initialize + tools/list` evidence exists, not just a package name or config entry.
- **Right:** sensitive/stateful/external servers stay serialized/session/project-isolated until stronger evidence exists.
- **Wrong:** UI claims a server is usable while it is disabled, untested, missing launch metadata, or blocked by policy/source mismatch.
- **Wrong:** a pasted command can chain shell operations, leak env/header values, or auto-enable tools without explicit Test.
- **Wrong:** green Node tests are treated as production proof without Rust build and one live MCP client call.
