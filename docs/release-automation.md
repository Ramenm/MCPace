# Release Automation and Update Policy

MCPace release delivery has three separate concerns:

1. **Auto-start** — runtime clients may launch MCPace on demand through the configured command or an explicit user-installed service.
2. **Auto-update** — users update through the installation channel (`npx`, `npm install -g`, GitHub Release assets, or a future package manager integration).
3. **Auto-release** — CI builds, verifies, attests, packs, and publishes artifacts from a protected tag.

The Rust binary does not silently rewrite itself. The safe runtime command is `mcpace update check`, which reports the current version, an optional latest version, and recommended package-manager commands.

## Single source of truth

`release-targets.json` owns the supported and planned target list. The following generated or verified surfaces must stay aligned with it:

- `packages/npm/cli/lib/targets.js`
- `packages/npm/cli/package.json` optional dependencies
- `packages/npm/cli-*/package.json` platform filters
- `scripts/github-release-matrix.mjs` GitHub Actions native matrix output
- `.github/workflows/release.yml` generated native release matrix consumer
- `.github/workflows/release-dry-run.yml` generated native proof matrix consumer

Run:

```sh
npm run sync:platform-packages
npm run verify:release-targets
npm run generate:release-matrix
npm run verify:platform-packages
npm run test:rust:ci
```

Native GitHub Actions matrices are generated by `scripts/github-release-matrix.mjs` from `release-targets.json`; workflows consume the generated `native_matrix` output with `fromJson(...)` instead of duplicating platform rows inline. Rust tests are executed by `scripts/run-rust-tests.mjs` one suite at a time with a per-suite timeout. Default CI is budget-first: Ubuntu carries the broad source/Rust gates, Windows keeps a targeted launcher smoke, and expensive hosted platform/Docker proof lanes run only when `workflow_dispatch` sets `full_ci: true` or a pull request is labeled `full-ci`. Release dry-runs and tag releases still run native host proof across Linux, Windows, and macOS.

## Release stages

### 1. Dry run, no publishing

`release-dry-run.yml` is manual-only to avoid spending macOS/Windows minutes on every pull request. Run it before creating a release tag when you need full source bundle, native binary build matrix, vendored binary lane, platform package lane, and npm pack dry-run proof without creating public releases.

### 2. Draft GitHub Release

`release.yml` runs on `v*.*.*` tags or manual dispatch. It builds all supported native targets, creates npm tarballs, generates checksums, and creates or updates a draft GitHub Release. Attestations are generated when supported by the repository visibility and account plan.

### 3. npm Trusted Publishing

`publish-npm.yml` is manual and tag-gated. It downloads prebuilt `.tgz` files and `SHA256SUMS.txt` from the matching GitHub Release, verifies every downloaded npm tarball against the release checksum manifest, injects the exact GitHub repository metadata required by npm Trusted Publishing, verifies package metadata, and publishes platform packages before the thin launcher.

The workflow defaults to `dry_run: true`.

## Supported native npm packages

- `@mcpace/cli-linux-x64-gnu`
- `@mcpace/cli-linux-arm64-gnu`
- `@mcpace/cli-darwin-x64`
- `@mcpace/cli-darwin-arm64`
- `@mcpace/cli-win32-x64-msvc`
- `@mcpace/cli-win32-arm64-msvc`

Planned but not published until separately proved:

- `@mcpace/cli-linux-x64-musl`
- `@mcpace/cli-linux-arm64-musl`

## Update behavior

`mcpace update check` supports these safe modes:

```sh
mcpace update check --json
mcpace update check --json --latest-version 0.4.1
MCPACE_LATEST_VERSION=0.4.1 mcpace update check --source env --json
MCPACE_UPDATE_SOURCE=npm mcpace update check --source npm --json
```

The command never mutates the binary. The recommended update path is external and explicit:

```sh
npm install -g @mcpace/cli@latest
npx @mcpace/cli@latest help
```

## Auto-start behavior

MCPace supports on-demand startup through client configuration and `stdio-shim`/hub lifecycle surfaces. Long-running OS service installation remains opt-in and should be implemented per platform later:

- macOS: LaunchAgent
- Linux: systemd user service
- Windows: Scheduled Task or Service

Default behavior should remain on-demand, not a permanently installed daemon.
